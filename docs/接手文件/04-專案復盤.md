## 復盤

### 做得好

- 使用 Prisma ORM 讓資料庫切換（SQLite ↔ PostgreSQL）非常順暢
- 專案管理框架有幫助到流程結構化
- VPS 上用 Docker 跑 PostgreSQL 讓部署和日後搬遷都很乾淨
- Next.js Server Components 讓頁面渲染和資料查詢整合得很好

### 做得不好

- PowerShell 的 `@"..."@` 字串會吃掉反引號 `` ` `` 和展開 `${}`，導致大量轉義問題
- Prisma 7 的 schema engine 在本機有相容性問題，需降版到 Prisma 6
- 一開始沒先確認伺服器 SSH 連線狀況，花了不少時間除錯
- Fine-grained GitHub PAT 沒有建立 repo 的權限，流程卡住

### 改進方案

| 問題 | 改進做法 |
|------|----------|
| PowerShell 轉義問題 | 一律使用 `[System.IO.File]::WriteAllText` 寫檔 |
| Prisma 版本 | 先在本地測試 schema engine 相容性再選版本 |
| SSH 連線 | 先確認防火牆和 SSH 服務再開始 |
| GitHub Token | 使用 classic PAT 並確保 `repo` scope |

## 技術債 / 待解決問題

- [ ] PostCSS config（`postcss.config.mjs`）可能不是 Tailwind CSS v4 + Next.js 16 的必要配置，需確認是否能移除
- [ ] `.env` 包含本地 SQLite 路徑，部署到 Vercel 時會發出警告
- [ ] `known_hosts.old` 有舊的 SSH 主機指紋待清理
- [ ] Docker 容器 `iti-postgres` 無持久化 volume 掛載（需加入 volume 設定）

## 關鍵經驗

1. **Prisma provider 無法透過環境變數動態切換** — provider 是 schema 中的靜態值，SQLite 和 PostgreSQL 需手動切換
2. **Vercel + PostgreSQL** — 透過 Docker 在 VPS 上跑 PostgreSQL 比用 Neon/Managed 服務更符合台灣用戶的延遲需求
3. **PowerShell 檔案寫入** — 永遠使用 `[System.IO.File]::WriteAllText(path, content)`，避免 `Out-File` 的 BOM 問題和 `@"..."@` 的轉義問題
4. **Vercel CLI env add** — 語法為 `vercel env add <name> <environment> --value <value> --yes`
